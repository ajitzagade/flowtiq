## Codebase Audit Findings

### F-01 — Login: Hardcoded branding breaks white-label promise

**File:** `apps/admin-portal/src/app/(auth)/login/page.tsx`
The left panel uses the literal string "Flowtiq" and a hardcoded `<Layers>` icon. Demo stat numbers (24 projects, 8 follow-ups, 142 documents) are static. For a white-label product where Vastudeep Associates gets their own app, this is a broken promise — the login page doesn't reflect tenant branding.
**Severity:** High — visible to every user on every login.

### F-02 — Dashboard: Bottom summary row duplicates top stat cards

**File:** `apps/admin-portal/src/app/(dashboard)/dashboard/page.tsx:471-499`
The bottom row shows "Completed Projects", "Projects On Hold", and "Total Follow-ups" as separate stat cards. The top row already shows "Active Projects" and "Pending Follow-ups". The duplication adds visual noise without adding information density. "Total Follow-ups" at the bottom doesn't differentiate from "Pending Follow-ups" at the top.
**Severity:** Medium — reduces information hierarchy.

### F-03 — Dashboard: Upcoming Follow-ups not navigable

**File:** `dashboard/page.tsx:413-423`
Follow-up items in the "Upcoming Follow-ups" panel are plain `<div>` elements with no `href` or click handler. Users can see the follow-up but cannot navigate to it.
**Severity:** High — breaks the expected "click to act" mental model.

### F-04 — Dashboard: Recent Activity has no "View all" link

**File:** `dashboard/page.tsx:427-461`
Recent Activity is capped at 8 items with `slice(0, 8)` and has no link to the full audit log.
**Severity:** Low — users who need the full log will find the Audit Logs nav item.

### F-05 — Dashboard: Active Projects alternating row colors are inconsistent

**File:** `dashboard/page.tsx:354-363`
Odd rows use `bg-white hover:bg-indigo-50`, even rows use `bg-violet-50 hover:bg-violet-100`. This violet/indigo split reads as an accidental inconsistency, not intentional design. The hover states use different color families.
**Severity:** Low — visual inconsistency, not a functional gap.

### F-06 — Header: Bell has no inline dropdown preview

**File:** `apps/admin-portal/src/components/layout/Header.tsx`
The notification bell navigates full-page to `/notifications` instead of showing a dropdown preview. Users lose context when checking notifications while in the middle of a task.
**Severity:** High — disrupts workflow. Industry standard is a popover with recent items.

### F-07 — Header: Avatar links to Settings with no dropdown

**File:** `Header.tsx:60-65`
The user avatar `<Link href="/settings">` navigates to the Settings page. There is no user menu (profile, logout shortcut). The logout button is hidden at the bottom of the sidebar, which on mobile means opening the drawer to sign out.
**Severity:** Medium — discoverability issue. Logout is not obvious.

### F-08 — Header: No global search

**File:** n/a (feature absent)
No search functionality exists anywhere in the app. Finding a specific project requires navigating to Projects and using the page-level search box. Documents, follow-ups, and users each have their own search, but there is no unified search across entities.
**Severity:** Medium — power users and admins with large datasets will feel this.

### F-09 — Sidebar: Collapse affordance is too subtle

**File:** `Sidebar.tsx:143-150`
The sidebar collapse/expand button is a 16px `ChevronLeft`/`ChevronRight` icon with no label, no tooltip on desktop (tooltip only shows when collapsed). The click target is very small.
**Severity:** Low — usability friction, not blocking.

### F-10 — Sidebar: "Reports" permission mismatch

**File:** `Sidebar.tsx:18`
Reports is gated behind `roles:manage` permission — an admin-only permission. Analytically, Reports should be accessible to Project Managers who want to see project performance. This is likely a placeholder assignment that was never revisited.
**Severity:** Medium — PMs cannot access Reports.

### F-11 — Project Detail: All stage cards collapsed by default

**File:** `projects/[id]/page.tsx:262`
`const [expanded, setExpanded] = useState(false)` means every StageCard starts collapsed. Users arriving at a project detail page see a list of collapsed accordions with no visible status information. The status icon and badge are only visible when expanded.
**Severity:** High — first impression on the most-used page is a collapsed wall of cards.

### F-12 — Project Detail: No breadcrumb navigation

**File:** `projects/[id]/page.tsx`
The project detail page has a back `<ArrowLeft>` button (`/projects`) but no breadcrumb trail. Users cannot tell if they navigated here from Dashboard → Active Projects, or from the Projects list page. The context of how they arrived is lost.
**Severity:** Medium — navigational confusion, especially when arriving via deep link.

### F-13 — Project Detail: Inline stage update form UX is confusing

**File:** `projects/[id]/page.tsx — StageCard component`
The stage update form is revealed by `setShowUpdateForm(true)` inside the same collapsed card. When expanded, the form appears as a continuation of the card content, blurring the line between "read mode" and "edit mode". There is no clear "Save / Cancel" CTA placement.
**Severity:** Medium — edit-mode confusion leads to accidental unsaved changes.

### F-14 — Notifications: Items have no entity link

**File:** `notifications/page.tsx`
Notification items display type icon, message, and relative time — but no link to the entity they reference. A "Stage assigned" notification doesn't link to the stage. Users must manually navigate to find what was notified.
**Severity:** High — breaks the core value proposition of notifications.

### F-15 — Forms: No focus trap in modals

**File:** `follow-ups/page.tsx, documents/page.tsx, etc.`
All modals use `role="dialog" aria-modal="true"` but do not implement a focus trap. Tab key can escape the modal and interact with the obscured page behind it.
**Severity:** Medium — accessibility issue (WCAG 2.1 AA failure).

### F-16 — Settings: Direct `fetch()` bypasses auth interceptor

**File:** `settings/page.tsx:65-72`
The branding save mutation uses raw `fetch()` with a manually constructed `Authorization` header instead of the `api.patch()` helper. This bypasses the axios interceptor that handles 401/token refresh — if the token expires during a settings save, it silently fails instead of refreshing.
**Severity:** Medium — silent auth failure under race condition.

### F-17 — Mobile: Subtitle hidden at sm breakpoint

**File:** `Header.tsx:40`
`subtitle && <p className="... hidden sm:block">` — the page subtitle is hidden on small screens. "Welcome back, Ajit. Here is what is happening." disappears on phones. At tablet (sm) the user sees only the page title.
**Severity:** Low — cosmetic, not blocking.

### F-18 — Login: No "Forgot password" flow

**File:** `login/page.tsx`
The login form has no "Forgot password?" link. For a commercial SaaS product in active deployment, this is a significant omission — admins must reset passwords manually via the API.
**Severity:** High — customer-facing gap once Vastudeep goes live.

### F-19 — General: No empty states on several pages

Several pages lack illustrated empty states: Notifications (no items), Documents (no items), Audit Logs. They fall back to just showing nothing or an empty table.
**Severity:** Low — polish gap.

### F-20 — General: No keyboard shortcuts

No keyboard shortcuts exist for common actions (new project, search, toggle sidebar). This limits power user efficiency.
**Severity:** Low — nice-to-have, not a blocker.

## Extended Audit Findings (Full Codebase — Round 2)

_All remaining pages (follow-ups, documents, users, roles, workflows, audit-logs, reports, settings full read, notifications full read, projects/[id] full read, projects/page full read, layout, providers, BrandingApplicator) reviewed._

### F-21 — Settings: General + Notifications tabs are UI stubs

**File:** `settings/page.tsx:344-413`
The General tab has timezone/date-format `<select>` elements with no `value` binding, no state, and the Save button has no `onClick`. The Notifications tab renders 5 toggles all `defaultChecked` with no state management — clicking Save does nothing. These are cosmetic placeholders with no API integration.
**Severity:** High — settings silently don't save; users think they configured something.

### F-22 — Settings: `secondaryColor` collected but never applied

**File:** `settings/page.tsx:294-302` + `BrandingApplicator.tsx`
The Branding tab collects `secondaryColor` and saves it to the DB via the branding PATCH. However, `BrandingApplicator.tsx` only reads `primaryColor` — it never applies `secondaryColor` to any CSS variable. The setting has no effect on the UI.
**Severity:** Medium — dead setting that misleads users.

### F-23 — Design system: `violet-*` and `indigo-*` used inconsistently across the app

**Files:** `documents/page.tsx:439`, `reports/page.tsx:363`, `projects/page.tsx:35`, `workflows/page.tsx:23`, `lib/utils.ts:49`
DESIGN.md explicitly states: "Don't use `violet-*` and `indigo-*` interchangeably — pick one accent family per surface." Violations found:
- `documents/page.tsx`: `GitBranch` icon uses `text-violet-500` for workflow hierarchy level
- `reports/page.tsx`: Active KPI card uses `color="bg-violet-500"`
- `projects/page.tsx` PALETTE constant: contains `#6366f1` (indigo) and `#8b5cf6` (violet) together
- `workflows/page.tsx` STAGE_COLORS: contains `#8b5cf6` (violet) alongside `#3b82f6` (blue)
- `lib/utils.ts` getAvatarColor: rotates through `bg-blue-500`, `bg-violet-500`, `bg-emerald-500`, `bg-amber-500`, `bg-rose-500`, `bg-cyan-500`, `bg-indigo-500`, `bg-teal-500` — 5 of 8 colors are outside the design system palette
**Severity:** Medium — systematic color drift that makes the app feel incoherent over time.

### F-24 — Reports: Out-of-palette colors for KPI icons

**File:** `reports/page.tsx:361-383`
KPI icon backgrounds use: `bg-blue-500`, `bg-emerald-500`, `bg-violet-500`, `bg-red-500`, `bg-sky-500`, `bg-teal-500`, `bg-amber-500`, `bg-slate-400`. Of these, `bg-sky-500` and `bg-teal-500` have no semantic meaning in the design system and are not defined as status colors.
**Severity:** Low — visual inconsistency on the Reports page only.

### F-25 — Global: All loading states use spinner — no skeleton loaders

**Files:** All 10 data pages
DESIGN.md states: "Show a skeleton loader for every async surface with more than 2 fields." The entire app uses a single animated SVG spinner (`animate-spin`) in table cells or centered in cards. No page implements skeleton loaders. This is a systematic gap — the spinner provides no spatial preview of incoming content.
**Severity:** Medium — missing pattern called out in the design spec.

### F-26 — Global: `confirm()` dialogs for destructive actions — 5 pages

**Files:** `documents/page.tsx:195`, `users/page.tsx:190`, `roles/page.tsx:226`, `workflows/page.tsx:349`, and `projects/page.tsx`
All destructive actions (delete document, hard-delete user, delete role, delete workflow) use the browser-native `window.confirm()` dialog. This is unstyled, cannot be customized, and blocks the main thread. DESIGN.md defines `.modal-overlay` / `.modal-content` classes — a confirmation modal using those classes would be consistent.
**Severity:** Low — functional but visually inconsistent with the rest of the UI.

### F-27 — Global: Table dates do not follow `text-right font-mono` alignment rule

**Files:** `follow-ups/page.tsx`, `audit-logs/page.tsx`, `users/page.tsx`, `documents/page.tsx`
DESIGN.md specifies: "Align all table dates right (`text-right font-mono`) for scannability." Dates in Follow-ups, Audit Logs, Users (Last Login, Joined), and Documents use `text-sm text-slate-500` or `text-sm text-slate-600` without `text-right` or `font-mono`. Only the IP address column in audit-logs correctly uses `font-mono`.
**Severity:** Low — minor scannability regression across all table pages.

### F-28 — app/layout.tsx: Page `<title>` hardcoded as "Flowtiq | Workflow Management"

**File:** `apps/admin-portal/src/app/layout.tsx:7`
`metadata.title = 'Flowtiq | Workflow Management'` — this is the browser tab title visible to every user. For a white-label product where Vastudeep Associates gets their own portal, the tab should read "Vastudeep | Workflow Management". This requires dynamic metadata generation from the tenant slug (available at request time via hostname).
**Severity:** Medium — visible brand mismatch for white-label tenants; extends F-01.

### F-29 — BrandingApplicator: `adjustHex` may produce invalid colors for dark primaries

**File:** `BrandingApplicator.tsx:12-18`
`adjustHex(primary, -20)` subtracts 20 from each R/G/B channel to generate a hover color. For a dark primary (e.g. `#0a0a0a`), this underflows to `#000000` — not a useful hover variant. For a light primary (e.g. `#f0f0f0`), the button text (always white) becomes illegible. The function has no awareness of luminance; a proper HSL-based darken would be safer.
**Severity:** Low — only affects tenants who set unusual primary colors.

### F-30 — Dashboard layout: `bg-slate-50` hardcoded instead of CSS variable

**File:** `apps/admin-portal/src/app/(dashboard)/layout.tsx:40`
`<div className="flex min-h-screen bg-slate-50">` — the body background is hardcoded as Tailwind's `bg-slate-50`. The CSS variable `--body-bg` is defined in DESIGN.md but not actually declared in `globals.css` or used in the layout. If a tenant overrides the body background via `BrandingApplicator`, it has no effect because the layout uses a static class.
**Severity:** Low — limits tenant background customization.

### F-31 — Providers: Toast background hardcodes `#1e293b`

**File:** `apps/admin-portal/src/app/providers.tsx:29`
The global toast config uses `background: '#1e293b'` (Tailwind slate-800) as an inline style. This is a hardcoded value that cannot be overridden via CSS variables. It also doesn't match any token in DESIGN.md. Acceptable as-is for a dark notification pill, but should be documented as intentional.
**Severity:** Low — cosmetic; consistent across all pages.

### F-32 — Settings: Branding save + remove-logo both bypass api helper (raw fetch)

**File:** `settings/page.tsx:65-72, 129-136`
Previously logged as F-16 (branding save). The full read reveals `handleRemoveLogo` also uses raw `fetch()` with a manually constructed `Authorization` header (lines 129-136). Both the save and remove logo paths bypass the axios interceptor. Two functions need to be migrated to `api.patch()`.
**Severity:** Medium — same silent auth-failure risk as F-16 (extends it).

### F-33 — Notifications page: Click-to-read marks read but does NOT navigate

**File:** `notifications/page.tsx:113`
`onClick={() => !notif.isRead && markReadMutation.mutate(notif.id)}` — clicking a notification item only marks it read. There is no navigation to the related entity, and no `entityId`/`entityType`/`entityUrl` field displayed or used. This confirms F-14 at the full-page level too (not just the popover).
**Severity:** High — same as F-14; confirming it affects the full page, not just a missing popover feature.

### F-34 — Documents: Workflow icon uses `text-violet-500` (palette violation)

**File:** `documents/page.tsx:439`
`<GitBranch size={15} className="text-violet-500 flex-shrink-0" />` for the workflow sub-header icon. The project level uses `text-blue-500` (FolderOpen), stage level uses `text-amber-500` (Layers). The violet choice here is arbitrary and not in the semantic color map.
**Severity:** Low — part of F-23 but worth isolating for the fix PR.

## Design Decisions

| # | Decision | Rationale |
|---|---|---|
| D-01 | Retain existing color tokens for the revamp — no palette change | The navy/blue palette is solid; issues are structural, not chromatic |
| D-02 | Notification bell → popover/dropdown, not full-page navigation | Preserves workflow context; industry standard |
| D-03 | Avatar → dropdown menu with profile + logout | Logout discoverability + settings access |
| D-04 | Stage cards: first card expanded by default, rest collapsed | Balances information density with progressive disclosure |
| D-05 | Notification items: each has a clickable entity link | Core value proposition of notifications |
| D-06 | Login left panel: use tenant branding dynamically (read from API pre-auth) | Matches white-label promise; tenant slug known from domain |
| D-07 | Breadcrumb: add to project detail and other deep pages | Navigational orientation |
| D-08 | Focus trap: implement via `@headlessui/react` Dialog or custom hook | WCAG 2.1 AA compliance |
| D-09 | Settings: replace raw fetch with api.patch helper | Consistent auth handling |
| D-10 | Bottom summary row on dashboard: replace with trend indicators on top stat cards | Reduces duplication, increases information density |
| D-11 | Follow-up items on dashboard: make clickable with Link to /follow-ups?id=xxx | Actionable dashboard |
| D-12 | Reports permission: change to 'reports:view' accessible to Project Managers | Aligns with existing seed data permission |
| D-13 | Settings General + Notifications tabs: wire to API — not UI stubs | F-21; stubs break trust in settings |
| D-14 | secondaryColor: apply to sidebar-text or a documented CSS variable | F-22; setting must have visible effect |
| D-15 | Color palette: standardise avatar/stage color pools to design-system tokens only | F-23; remove violet/indigo/teal/sky/rose/cyan from getAvatarColor and STAGE_COLORS constants |
| D-16 | Loading states: replace spinner with skeleton loaders for all async surfaces with >2 fields | F-25; DESIGN.md requirement |
| D-17 | Destructive confirm dialogs: replace window.confirm() with modal-overlay confirm pattern | F-26; consistent with existing modal system |
| D-18 | Table dates: add text-right font-mono to all date/time columns across table pages | F-27; DESIGN.md spec |
| D-19 | app/layout.tsx title: make tenant-aware via generateMetadata + hostname lookup | F-28; extends white-label fix F-01 |
| D-20 | Settings raw fetch: migrate handleRemoveLogo to api.patch() (extends D-09) | F-32 |
